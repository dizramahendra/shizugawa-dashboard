## Confirmed product structure

The experience starts from a shared Map Viewport.

From the map viewport, the user can select one of two water-body types:

### Path 1 — River
- User selects a river from the map viewport
- This opens a River Playback View (2D)
- The River Playback View should be parallel in structure to the Ocean Playback View, but adapted for 2D
- It should use the same temporal playback logic:
  - values change over time at fixed cells
  - no particle animation
  - playback-first interaction
- It should use a 2D pixel / gridded visualization
- It should not use depth representation

### Path 2 — Ocean
- User selects the ocean basin from the map viewport
- This opens an Ocean Playback View (3D)
- Keep the current 3D gridded / voxel ocean-basin model direction
- Do not remove or change the current depth section in the Ocean Playback View
- Leave the depth section as it is for now

## Design / UX intent

This is a 3D Time-Series / temporal playback product.

Shared logic across both River and Ocean playback views:
- playback over time is the primary interaction
- values change over time at fixed spatial cells
- no particle movement
- no decarbonization
- no comparison
- desktop only

The two playback views should feel like parallel sibling views in the same product family:
- same visual language
- same playback bar logic
- same variable-switching logic
- similar sidebar structure

But they differ in representation:
- River Playback View = 2D
- Ocean Playback View = 3D
